## 1. 项目概述

### 1.1 项目定位

RiskFlow 是一个**高性能、可扩展的实时风控决策平台**，基于 LiteFlow 流程编排引擎构建，提供灵活的规则配置、AI 智能分析和可视化编排能力。

### 1.2 核心特性

- **流程编排**：基于 LiteFlow 的 EL 表达式，支持串行、并行、条件分支
- **组件化设计**：风控节点组件化，易于扩展和复用
- **规则引擎**：支持动态规则配置和热更新
- **AI 增强**：集成 LLM 进行智能风险分析
- **多种部署模式**：支持独立服务、SDK 嵌入两种模式
- **高性能**：并行执行、缓存优化、异步处理

### 1.3 应用场景

- **登录风控**：异常登录检测、账号保护
- **支付风控**：交易反欺诈、资金安全
- **注册风控**：防刷注册、垃圾账号识别
- **内容风控**：违规内容识别、用户行为分析

## 3. 模块设计

### 3.1 risk-flow-core (核心引擎模块)

**职责**：提供风控流程编排的核心能力，可独立嵌入到其他项目中。

#### 3.1.1 目录结构

```
risk-flow-core/
├── component/                          # LiteFlow 组件
│   ├── base/                          # 基础检测组件
│   │   ├── LoadContextComponent       # 上下文加载
│   │   └── IpBlacklistCheckComponent  # IP黑名单检测
│   ├── rule/                          # 规则执行组件
│   │   └── RuleExecuteComponent       # 规则引擎
│   ├── ai/                            # AI分析组件
│   │   └── LlmRiskAnalyzeComponent    # LLM智能分析
│   ├── decision/                      # 决策组件
│   │   └── DecisionJudgeComponent     # 决策判定
│   └── post/                          # 后处理组件
│       └── DummyNodeComponent         # 空操作节点
├── context/                           # 上下文定义
│   └── RiskFlowContext                # 风控上下文
├── service/                           # 服务接口
│   ├── BlacklistService               # 黑名单服务接口
│   └── RuleService                    # 规则服务接口
└── resources/
    └── flow/
        └── risk-decision.xml          # 流程定义文件
```

#### 3.1.2 核心组件说明

| 组件名 | 节点ID | 功能描述 | 输入 | 输出 |
|--------|--------|----------|------|------|
| LoadContextComponent | loadContext | 初始化上下文，生成事件ID | RiskFlowContext | eventId, requestTime |
| IpBlacklistCheckComponent | ipBlacklistCheck | IP黑名单检测 | userIp | baseCheckResults, ruleScores |
| RuleExecuteComponent | ruleExecute | 执行业务规则 | eventType, features | ruleScores, totalRiskScore |
| LlmRiskAnalyzeComponent | llmRiskAnalyze | LLM智能分析 | context全量数据 | aiAnalysisResult |
| DecisionJudgeComponent | decisionJudge | 最终决策判定 | totalRiskScore | result, resultMessage |
| DummyNodeComponent | dummyNode | 空操作占位 | - | - |

#### 3.1.3 RiskFlowContext 设计

```java
public class RiskFlowContext implements Serializable {
    // ========== 输入参数 ==========
    private Long workflowId;              // 流程ID
    private String eventId;               // 事件ID
    private String eventType;             // 事件类型 (login/payment/register)
    private String userId;                // 用户ID
    private String userIp;                // 用户IP
    private String deviceId;              // 设备ID
    private Map<String, Object> features; // 特征数据
    private Map<String, Object> extInfo;  // 扩展信息
    
    // ========== 中间结果 ==========
    private Map<String, Boolean> baseCheckResults;  // 基础检测结果
    private Map<String, Integer> ruleScores;        // 规则得分
    private Integer totalRiskScore;                 // 总风险评分
    private String aiAnalysisResult;                // AI分析结果
    private Boolean isHighRisk;                     // 是否高风险
    
    // ========== 输出结果 ==========
    private DecisionResult result;        // 决策结果 (ACCEPT/REJECT/REVIEW/CHALLENGE)
    private String resultMessage;         // 决策消息
    private LocalDateTime decisionTime;   // 决策时间
    private Long executionTimeMs;         // 执行耗时
    private Map<String, Object> nodeResults; // 节点执行结果
}
```

## 4. 核心流程

### 4.1 流程编排设计

#### 4.1.1 LiteFlow 流程定义

```xml
<flow>
    <chain name="riskDecisionFlow">
        THEN(
            loadContext,                    <!-- 1. 加载上下文 -->
            parallelBaseCheck,              <!-- 2. 并行基础检测 -->
            ruleExecute,                    <!-- 3. 规则执行 -->
            IF(isHighRisk,                  <!-- 4. 条件判断 -->
                THEN(llmRiskAnalyze),       <!-- 高风险：AI分析 -->
                THEN(skipAiAnalyze)         <!-- 低风险：跳过AI -->
            ),
            decisionJudge,                  <!-- 5. 决策判定 -->
            POST(logRecord)                 <!-- 6. 后置：记录日志 -->
        );
    </chain>

    <chain name="parallelBaseCheck">
        WHEN(
            ipBlacklistCheck                <!-- IP黑名单检测 -->
            <!-- 未来可添加更多并行检测 -->
        );
    </chain>
</flow>
```

#### 4.1.2 流程执行图

```
┌─────────────────────────────────────────────────────────┐
│                    请求进入                              │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  Step 1: loadContext (上下文加载)                        │
│  - 生成 eventId                                          │
│  - 设置 requestTime                                      │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  Step 2: parallelBaseCheck (并行基础检测)                │
│  ┌─────────────────────────────────────────────────┐   │
│  │  ipBlacklistCheck (IP黑名单检测)                 │   │
│  │  - 检查 IP 是否在黑名单                          │   │
│  │  - 命中则加 50 分                                │   │
│  └─────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  Step 3: ruleExecute (规则执行)                          │
│  - 根据 eventType 加载对应规则                           │
│  - 评估规则表达式                                        │
│  - 累加风险评分                                          │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  Step 4: IF(isHighRisk) (条件分支)                       │
│                                                          │
│  totalRiskScore >= 50?                                   │
│         ↓ YES                    ↓ NO                    │
│  ┌──────────────┐         ┌──────────────┐             │
│  │llmRiskAnalyze│         │skipAiAnalyze │             │
│  │(AI智能分析)  │         │(跳过AI分析)  │             │
│  └──────────────┘         └──────────────┘             │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  Step 5: decisionJudge (决策判定)                        │
│  - >= 80: REJECT (直接拒绝)                              │
│  - >= 50: REVIEW (人工审核)                              │
│  - >= 30: CHALLENGE (验证码挑战)                         │
│  - < 30:  ACCEPT (审核通过)                              │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│  Step 6: POST(logRecord) (后置日志记录)                  │
│  - 保存决策日志到数据库                                  │
└─────────────────────────────────────────────────────────┘
                            ↓
┌─────────────────────────────────────────────────────────┐
│                    返回决策结果                          │
└─────────────────────────────────────────────────────────┘
```

### 4.2 决策阈值设计

| 风险评分 | 决策结果 | 说明 | 后续动作 |
|---------|---------|------|---------|
| >= 50 | REVIEW | 人工审核 | 进入审核队列，通知审核人员 |
| >= 30 | CHALLENGE | 验证码挑战 | 要求用户完成验证 |
| < 30 | ACCEPT | 审核通过 | 放行操作 |
| >= 80 | REJECT | 直接拒绝 | 阻断操作，记录日志 |

### 4.3 规则评分机制

#### 4.3.1 评分累加规则

```
总风险评分 = IP黑名单得分 + 规则1得分 + 规则2得分 + ... + 规则N得分
```

#### 4.3.2 示例规则配置

| 规则编码 | 规则名称 | 表达式 | 得分 | 适用场景 |
|---------|---------|--------|------|---------|
| login_new_device | 新设备登录 | isNewDevice == true | 20 | login |
| login_night_time | 深夜登录 | loginHour >= 0 && loginHour < 6 | 15 | login |
| login_failed_attempts | 登录失败次数过多 | failedLoginCount >= 3 | 30 | login |
| payment_large_amount | 大额支付 | amount > 10000 | 40 | payment |
| payment_new_payee | 新收款人 | isNewPayee == true | 25 | payment |

## 6. 技术选型

### 6.1 核心技术栈

| 技术 | 版本 | 用途 |
|------|------|------|
| Java | 21 | 开发语言 |
| Spring Boot | 3.2.0 | 应用框架 |
| LiteFlow | 2.15.3 | 流程编排引擎 |
| PostgreSQL | 14+ | 关系数据库 |
| Spring Data JPA | 3.2.0 | ORM框架 |
| Lombok | 1.18.30 | 代码简化 |
| Maven | 3.8+ | 构建工具 |

### 6.2 技术选型理由

#### 6.2.1 为什么选择 LiteFlow？

1. **轻量级**：无需额外部署，嵌入式运行
2. **高性能**：支持并行执行，性能优异
3. **易扩展**：组件化设计，易于添加新节点
4. **表达力强**：EL 表达式支持复杂流程编排
5. **Spring 集成**：与 Spring Boot 无缝集成

#### 6.2.2 为什么选择 PostgreSQL？

1. **JSON 支持**：原生支持 JSONB 类型，适合存储动态特征数据
2. **性能优异**：支持索引优化、并发控制
3. **开源免费**：无商业授权限制
4. **扩展性强**：支持分区表、物化视图等高级特性

## 8. 扩展性设计

### 8.1 组件扩展

添加新的风控组件只需：

1. 创建新的 Component 类，继承 `NodeComponent`
2. 使用 `@LiteflowComponent` 注解标注节点ID
3. 实现 `process()` 方法
4. 在流程定义中引用该节点

示例：
```java
@LiteflowComponent("deviceFingerprintCheck")
public class DeviceFingerprintComponent extends NodeComponent {
    @Override
    public void process() {
        RiskFlowContext context = this.getContextBean(RiskFlowContext.class);
        // 实现设备指纹检测逻辑
    }
}
```

### 8.2 规则扩展

支持通过数据库动态配置规则，无需重启服务。

### 8.3 流程扩展

支持通过修改 XML 配置文件或数据库动态调整流程编排。
